# [프로젝트 기획서] 경량형 서버 상태 감시 시스템 구축

## 1. 개요 및 목적
* **목적:** 운영 중인 윈도우 서버의 자원 상태(CPU, Memory 등)를 실시간으로 감시하고, 이를 한눈에 볼 수 있는 웹 관제 대시보드를 구축한다.
* **기반 환경:** 대상 서버에는 이미 MS-SQL(메모리 약 4.3GB 점유), 다수의 Java 및 Node.js 서비스가 구동 중이므로, **기존 서비스에 영향을 주지 않는 초경량·고성능 시스템** 구축을 최우선으로 한다.

---

## 2. 시스템 아키텍처 (Architecture)

### 2.1 통신 및 데이터 흐름 방식: **Agent Push 방식**
* 각 감시 대상 서버에 설치된 에이전트가 중심 백엔드 서버로 데이터를 직접 던지는 구조를 채택한다.
* **장점:** 백엔드 서버의 포트 하나만 방화벽을 열면 되므로 보안 및 인프라 관리가 매우 단순해진다.

### 2.2 통신 프로토콜: **HTTP (REST API / JSON)**
* **이유:** 초단위 스트리밍이 아닌 5~10초 주기의 상태 수집이므로, 상시 연결을 유지해야 하는 WebSocket보다 **필요할 때만 요청하고 끊는 HTTP 방식**이 서버 자원(네트워크 소켓, 메모리) 관리 측면에서 압도적으로 효율적이다.

### 2.3 데이터 저장 방식: **SQLite (내장형 파일 DB)**
* MySQL 등 무거운 DB를 별도 설치하지 않고, `monitoring.db`라는 **단일 파일 형태의 내장 DB**를 사용한다.
* **장점:** 서버 리소스를 거의 먹지 않으면서도, SQL 쿼리를 통해 과거 특정 시간대의 자원 추이를 쉽게 조회하여 대시보드 그래프를 그릴 수 있다.

---

## 3. 기술 스택 (Tech Stack)

* **백엔드 (Server):** **Go (Golang)**
  * 가상 머신(VM) 없이 구동되어 메모리 사용량이 수십 MB 수준으로 매우 적음. 동시성 처리가 뛰어나 대량의 에이전트 요청 수집에 최적화됨.
* **에이전트 (Agent):** **Go (Golang)**
  * 백엔드와 언어를 통일하여 데이터 구조체 코드를 공유함. 빌드 시 `agent.exe` 파일 하나만 떨어지므로 대상 서버에 추가 런타임 설치 없이 단독 배포 가능.
* **프론트엔드 (UI):** **Go 내장 템플릿 (`html/template`)**
  * 초기 단계에서는 별도의 프론트엔드 빌드 도구(React 등) 없이, Go 백엔드가 직접 HTML/CSS 화면을 그려 전달하는 SSR(서버 사이드 렌더링) 방식을 채택하여 개발 속도를 극대화함.

---

## 4. 네트워크 및 포트 설계

* **서비스 포트:** **`9090` (TCP)**
  * **이유:** 기존 서버의 활성 포트(MS-SQL: 1488, Java: 8090/8910~12, Node: 5173~75)들과의 충돌을 완벽히 회피함. 또한 모니터링 업계 표준(Prometheus) 관례 포트를 따름으로써 인프라 직관성을 높임.
* **방화벽 설정:** 백엔드가 구동되는 서버의 윈도우 방화벽(Inbound)에서 `9090` 포트를 허용하여 외부 에이전트들이 접근할 수 있도록 조치함.

## 5. 소스코드 저장소(Git) 구조: **모노레포 (Monorepo)**

관리 효율성과 개발 속도를 위해 하나의 깃허브 레포지토리 안에서 서버와 에이전트 코드를 모두 관리한다.

* **레포지토리 이름:** `go-watchdog`
* **디렉토리 구조:**
```text
📁 go-watchdog (Repository Root)
 ├── 📁 common    # 서버와 에이전트가 공동으로 사용할 데이터 구조체 (Metric DTO)
 ├── 📁 server    # 9090 포트를 열고 대기하는 Go 백엔드 + HTML 대시보드 템플릿
 └── 📁 agent     # 각 서버에서 시스템 자원(CPU/RAM)을 수집해 보낼 Go 에이전트
