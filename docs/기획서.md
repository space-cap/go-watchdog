# [프로젝트 기획서] 경량형 서버 상태 감시 시스템 구축

## 1. 개요 및 목적
* **목적:** 운영 중인 윈도우 서버의 자원 상태(CPU, Memory 등)를 실시간으로 감시하고, 이를 한눈에 볼 수 있는 웹 관제 대시보드를 구축한다.
* **기반 환경:** 대상 서버에는 이미 MS-SQL(메모리 약 4.3GB 점유), 다수의 Java 및 Node.js 서비스가 구동 중이므로, **기존 서비스에 영향을 주지 않는 초경량·고성능 시스템** 구축을 최우선으로 한다.

---

## 2. 시스템 아키텍처 (Architecture)

### 2.1 통신 및 데이터 흐름 방식: **Agent Push 방식**
* 각 감시 대상 서버에 설치된 에이전트가 중심 백엔드 서버로 데이터를 직접 던지는 구조를 채택한다.
* **장점:** 백엔드 서버의 포트 하나만 방화벽을 열면 되므로 보안 및 인프라 관리가 매우 단순해진다.

### 2.2 통신 프로토콜: **HTTP (REST API / JSON)**
* **이유:** 초단위 스트리밍이 아닌 5~10초 주기의 상태 수집이므로, 상시 연결을 유지해야 하는 WebSocket보다 **필요할 때만 요청하고 끊는 HTTP 방식**이 서버 자원(네트워크 소켓, 메모리) 관리 측면에서 압도적으로 효율적이다.

### 2.3 데이터 저장 방식: **SQLite (내장형 파일 DB)**
* MySQL 등 무거운 DB를 별도 설치하지 않고, `monitoring.db`라는 **단일 파일 형태의 내장 DB**를 사용한다.
* 에이전트 다수의 동시 쓰기 시 발생할 수 있는 락(Lock) 이슈를 최소화하기 위해 **WAL(Write-Ahead Logging) 모드**를 활성화한다.
* **장점:** 서버 리소스를 거의 먹지 않으면서도, SQL 쿼리를 통해 과거 특정 시간대의 자원 추이를 쉽게 조회하여 대시보드 그래프를 그릴 수 있다.

### 2.4 데이터 보존 및 정리 정책 (Data Retention)
* **보존 정책:** 서버의 디스크 용량 잠식을 방지하기 위해 최대 14일(또는 30일) 간의 메트릭만 저장한다.
* **처리 방식:** 백엔드 서버에서 주기적인 백그라운드 고루틴(Goroutine)을 돌려 기준 기한이 지난 과거 데이터를 `DELETE` 쿼리로 자동 삭제한다.

### 2.5 에이전트 인증 및 보안 (Agent Authentication)
* **인증 방식:** 에이전트가 데이터를 백엔드로 전송할 때 HTTP 헤더에 사전에 약속된 **보안 토큰(API Key)**을 실어 전송한다.
* **이유:** 외부 또는 망 내 비인가 사용자가 무단으로 백엔드 API에 임의의 모니터링 데이터를 주입하거나 위조하는 것을 차단한다.

### 2.6 에이전트 오프라인 감지 (Heartbeat)
* **감시 로직:** 백엔드는 각 에이전트로부터 데이터를 마지막으로 수집한 시간(`Last Seen`)을 기록한다.
* **처리 방식:** 마지막 수집 시간이 약속된 주기(예: 10초)의 3배(30초) 이상 지연될 경우, 대시보드 화면상에서 해당 에이전트의 상태를 `OFFLINE` 또는 `경고`로 표시하여 장애 상황을 즉시 인지하도록 유도한다.

## 3. 기술 스택 (Tech Stack)

* **백엔드 (Server):** **Go (Golang)**
  * 가상 머신(VM) 없이 구동되어 메모리 사용량이 수십 MB 수준으로 매우 적음. 동시성 처리가 뛰어나 대량의 에이전트 요청 수집에 최적화됨.
* **에이전트 (Agent):** **Go (Golang)**
  * 백엔드와 언어를 통일하여 데이터 구조체 코드를 공유함. 빌드 시 `agent.exe` 파일 하나만 떨어지므로 대상 서버에 추가 런타임 설치 없이 단독 배포 가능.
* **프론트엔드 (UI):** **Go 내장 템플릿 (`html/template`) + Vanilla JS (AJAX Polling)**
  * SSR(서버 사이드 렌더링) 방식으로 개발하되, 실시간 모니터링 편의를 위해 클라이언트 사이드(브라우저)에서 Vanilla JS를 통해 5~10초 주기로 백엔드 API(`GET /api/status`)를 비동기 호출(Fetch)하여 대시보드를 새로고침 없이 갱신한다.

## 4. 네트워크 및 포트 설계

* **서비스 포트:** **`9090` (TCP)**
  * **이유:** 기존 서버의 활성 포트(MS-SQL: 1488, Java: 8090/8910~12, Node: 5173~75)들과의 충돌을 완벽히 회피함. 또한 모니터링 업계 표준(Prometheus) 관례 포트를 따름으로써 인프라 직관성을 높임.
* **방화벽 설정:** 백엔드가 구동되는 서버의 윈도우 방화벽(Inbound)에서 `9090` 포트를 허용하여 외부 에이전트들이 접근할 수 있도록 조치함.

## 5. 소스코드 저장소(Git) 구조: **모노레포 (Monorepo)**

관리 효율성과 개발 속도를 위해 하나의 깃허브 레포지토리 안에서 서버와 에이전트 코드를 모두 관리한다.

* **레포지토리 이름:** `go-watchdog`
* **디렉토리 구조:**
```text
📁 go-watchdog (Repository Root)
 ├── 📁 common    # 서버와 에이전트가 공동으로 사용할 데이터 구조체 (Metric DTO)
 ├── 📁 server    # 9090 포트를 열고 대기하는 Go 백엔드 + HTML 대시보드 템플릿
 └── 📁 agent     # 각 서버에서 시스템 자원(CPU/RAM/Disk)을 수집해 보낼 Go 에이전트
```

## 6. 에이전트 상세 및 Windows 환경 최적화

### 6.1 수집 대상 메트릭 확장
* CPU 사용률(%), 메모리 사용률(%) 외에도 실질적 장애 요인이 되는 각 드라이브(C:, D:)의 **디스크 남은 용량 및 총량(Disk Space)**을 함께 수집한다.

### 6.2 Windows 백그라운드 서비스 등록
* **실행 방식:** 에이전트가 윈도우 로그인 여부나 재부팅과 관계없이 항상 동작하도록 **윈도우 서비스(Windows Service)** 형태로 등록하여 관리한다. (NSSM 등의 도구 활용 또는 Go 내 svc 패키지를 활용한 서비스 빌드 고려)

### 6.3 에이전트 설정 관리 (Configuration)
* 에이전트 실행 파일과 동일한 경로상에 설정 파일(예: `config.json`)을 두어 백엔드 서버 IP 주소, 보안 토큰, 메트릭 수집 주기 등을 코드 컴파일 없이 동적으로 변경 가능하도록 한다.
